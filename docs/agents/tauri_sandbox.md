# Tauri Permission Files Under Bazel Sandbox

## Problem

`tauri_build::try_build()` fails to locate permission TOML files when
`dashql-native` is compiled under a Bazel sandbox, because the paths it
receives contain stale sandbox-specific absolute prefixes.

## Root Cause

Tauri plugins declare permissions via TOML files.  During a Cargo/Bazel build,
the `tauri-build` crate discovers them through `DEP_*_PERMISSION_FILES_PATH`
environment variables.  Each such variable points to a JSON index file whose
contents look like:

```json
["/private/var/folders/xyz/.../sandbox/12345/bazel-out/k8-opt/bin/.../permission_files.json"]
```

The JSON index files are generated by a previous Bazel action and baked with
the **absolute path of that action's sandbox**.  When the `cargo_build_script`
action for `dashql-native` runs in its own sandbox (a different ephemeral
directory), both the env var value and the paths inside the JSON point to a
non-existent location, causing `try_build()` to get `ENOENT` on every
permission TOML.

## Fix

`packages/dashql-native/build.rs` contains two helpers that remap stale paths
before calling `tauri_build::try_build()`.

### `remap_sandbox_path`

Replaces the absolute prefix before `bazel-out/` in a single path string with
the current exec root:

```rust
fn remap_sandbox_path(path: &str, exec_root: &str) -> String {
    match path.find("bazel-out/") {
        Some(i) => format!("{}{}", exec_root, &path[i..]),
        None => path.to_string(),
    }
}
```

### `remap_all_sandbox_paths`

Applies the same substitution to every `bazel-out/`-anchored path embedded
anywhere in an arbitrary text blob (e.g. raw JSON):

```rust
fn remap_all_sandbox_paths(text: &str, exec_root: &str) -> String { ... }
```

It scans for `bazel-out/` markers, walks back to the enclosing `"` to find the
path start, and splices in `exec_root` in place of the stale prefix.

### `rewrite_permission_env_vars`

Iterates over all `DEP_*_PERMISSION_FILES_PATH` env vars, remaps the JSON
index path itself, rewrites the JSON contents (all embedded paths), writes a
corrected copy to `OUT_DIR`, and updates the env var to point at the copy —
all before `try_build()` is invoked:

```rust
fn rewrite_permission_env_vars(exec_root: &str, out_dir: &str) { ... }
```

The current exec root is derived from `OUT_DIR`, which Bazel always sets to a
valid path in the active sandbox:

```rust
let exec_root = out_dir
    .find("bazel-out/")
    .map(|i| &out_dir[..i])
    .unwrap_or("");
```

## Why This Cannot Be Removed

The remapping logic is load-bearing as long as:

1. `rules_rust`/`cargo_build_script` does **not** remap `DEP_*` env vars
   through its own exec-root expansion before passing them to the build script.
2. `tauri-build` stores permission file paths as absolute (not
   `bazel-out/`-relative) strings inside the JSON index files.
3. No custom Bazel rule regenerates the `DEP_*` paths fresh for each sandbox
   invocation.

If any of the above changes upstream, the three helpers and the
`rewrite_permission_env_vars` call in `main()` can be removed.

## Files

| File | Role |
|---|---|
| `packages/dashql-native/build.rs` | Contains `remap_sandbox_path`, `remap_all_sandbox_paths`, `rewrite_permission_env_vars` |
| `packages/dashql-native/BUILD.bazel` | Passes `DEP_*_PERMISSION_FILES_PATH` via `link_deps` |
