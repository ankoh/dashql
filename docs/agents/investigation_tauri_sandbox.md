# Tauri Permission Files Under Bazel Sandbox

## Problem

`tauri_build::try_build()` fails to locate permission TOML files when
`dashql-native` is compiled under a Bazel sandbox.  There are **two distinct
path classes** that can be stale or inaccessible, each requiring a different
remapping strategy.

## Path Class 1 — Stale `bazel-out/` sandbox prefix

### Root Cause

Tauri plugins declare permissions via TOML files.  During a Cargo/Bazel build,
the `tauri-build` crate discovers them through `DEP_*_PERMISSION_FILES_PATH`
environment variables.  Each such variable points to a JSON index file whose
contents look like:

```json
["/private/var/folders/xyz/.../sandbox/12345/bazel-out/k8-opt/bin/.../permission_files.json"]
```

The JSON index files are generated by a previous Bazel action and baked with
the **absolute path of that action's sandbox**.  When the `cargo_build_script`
action for `dashql-native` runs in its own sandbox (a different ephemeral
directory), both the env var value and the paths inside the JSON point to a
non-existent location, causing `try_build()` to get `ENOENT` on every
permission TOML.

### Fix — `remap_sandbox_path` and `remap_all_sandbox_paths`

`remap_sandbox_path` replaces the absolute prefix before `bazel-out/` in a
single path string with the current exec root:

```rust
fn remap_sandbox_path(path: &str, exec_root: &str) -> String {
    match path.find("bazel-out/") {
        Some(i) => format!("{}{}", exec_root, &path[i..]),
        None => path.to_string(),
    }
}
```

`remap_all_sandbox_paths` applies the same substitution to every
`bazel-out/`-anchored path embedded anywhere in a raw JSON string.  It scans
for `bazel-out/` markers, walks back to the enclosing `"` to find the path
start, and splices in `exec_root` in place of the stale prefix.

## Path Class 2 — Cached `~/.cache/bazel-repo/contents/` paths

### Root Cause

When a plugin's build script output (including its `PERMISSION_FILES_PATH`
JSON) is **restored from the Bazel remote cache**, the TOML paths embedded
in that JSON look like:

```
/home/runner/.cache/bazel-repo/contents/<sha256>/<uuid>/permissions/autogenerated/commands/get_current.toml
```

The `<uuid>` component is specific to the runner that originally produced the
cache entry.  A different CI runner has a different UUID in its local
`bazel-repo` cache, so the files do not exist at those paths — `ENOENT`.

This path class is **not** caught by `remap_all_sandbox_paths` because it
contains no `bazel-out/` marker.

### Fix — `find_in_external`, `find_in_repo_cache`, and `remap_repo_cache_paths`

`remap_repo_cache_paths` walks the JSON string, finds every
`/bazel-repo/contents/` occurrence, extracts the relative path after
`<sha256>/<uuid>/`, and tries two strategies in order:

**Strategy 1 — `find_in_external`** (works when the TOML files are declared as
sandbox inputs).

The TOML files are external crate source files that may be accessible in the
sandbox under `<exec_root>/external/<crate-dir>/`.  The crate-relative path
(e.g. `permissions/autogenerated/commands/get_current.toml`) is preserved, so
we can probe `external/` for a matching file:

```rust
fn find_in_external(relative: &str, exec_root: &str) -> Option<String> {
    let external = std::path::Path::new(exec_root).join("external");
    for entry in std::fs::read_dir(&external).ok()?.flatten() {
        let candidate = entry.path().join(relative);
        if candidate.exists() {
            return Some(candidate.to_string_lossy().into_owned());
        }
    }
    None
}
```

**Strategy 2 — `find_in_repo_cache`** (fallback when Strategy 1 fails; handles
the stale-UUID scenario).

The macOS darwin-sandbox explicitly allows read access to `~/.cache/bazel-repo/`
(Bazel adds this to the sandbox profile).  The content hash in the path is
deterministic and stable across runners; only the UUID per-session changes.
By enumerating all UUID subdirectories under the same hash we can find the file
regardless of which runner originally wrote the JSON:

```rust
fn find_in_repo_cache(full_path: &str) -> Option<String> {
    const MARKER: &str = "/bazel-repo/contents/";
    let idx = full_path.find(MARKER)?;
    let prefix = &full_path[..idx + MARKER.len()];
    let after = &full_path[idx + MARKER.len()..];
    // after: <hash>/<uuid>/<relative>
    let mut parts = after.splitn(3, '/');
    let hash = parts.next()?;
    let _uuid = parts.next()?;
    let relative = parts.next()?;
    let hash_dir = std::path::PathBuf::from(format!("{}{}", prefix, hash));
    for entry in std::fs::read_dir(&hash_dir).ok()?.flatten() {
        let candidate = entry.path().join(relative);
        if candidate.exists() {
            return Some(candidate.to_string_lossy().into_owned());
        }
    }
    None
}
```

Both strategies are chained:

```rust
let remapped = relative
    .and_then(|r| find_in_external(r, exec_root))
    .or_else(|| find_in_repo_cache(full_path));
```

A `cargo:warning=ACL-DIAG: remap repo-cache {from} -> {to}` line is emitted on
success; `cargo:warning=ACL-DIAG: remap repo-cache FAILED for {path}` is
emitted when both strategies fail (previously this was silent, making diagnosis
hard).

## `rewrite_permission_env_vars`

Orchestrates both remapping passes for every `DEP_*_PERMISSION_FILES_PATH` env
var, writes a corrected JSON copy to `OUT_DIR`, and updates the env var to
point at the copy — all before `try_build()` is invoked:

```
original env var value
  → remap_sandbox_path       (fix bazel-out/ prefix on the JSON file path itself)
  → read JSON
  → remap_all_sandbox_paths  (fix bazel-out/ prefixes inside the JSON)
  → remap_repo_cache_paths   (fix bazel-repo/contents/ paths inside the JSON)
  → write fixed-<KEY> to OUT_DIR
  → setenv <KEY> = fixed file
```

The current exec root is derived from `OUT_DIR`:

```rust
let exec_root = out_dir
    .find("bazel-out/")
    .map(|i| &out_dir[..i])
    .unwrap_or("");
```

## Silent `try_build` Failure — Historical Bug

Before March 2026, `try_build` errors were swallowed with a `cargo:warning`
and the build continued.  When `try_build` failed, `acl-manifests.json` and
`capabilities.json` were never written to `OUT_DIR`.  `generate_context!()`
then embedded an **empty permission set**, meaning every IPC command (including
`plugin:updater|check`) was silently denied at runtime.  The broken binary was
cached in bazel-remote and served on all subsequent CI builds.

`try_build` now returns `Err`, causing the build script — and thus the entire
Bazel action — to fail loudly.  This prevents a broken binary from ever being
built or cached.

---

## Why This Cannot Be Removed

The remapping logic is load-bearing as long as:

1. `rules_rust`/`cargo_build_script` does **not** remap `DEP_*` env vars
   through its own exec-root expansion before passing them to the build script.
2. `tauri-build` stores permission file paths as absolute strings (not
   exec-root-relative) inside the JSON index files.
3. Plugin build script outputs are cached in bazel-remote and restored on different
   runners whose `bazel-repo` UUID differs from the originating runner.
